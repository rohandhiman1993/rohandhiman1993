# Module pattern

The module pattern ensures private and public encapsulatin in javascript protecting the global namespace and diminishing naming conflicts.

const ChickenModule = (() => {
  const eggColor = "white"; //private variable
  const chuck = () => console.log("chuck chuck"); // private function
  return {
    layEgg: () => {
      console.log("laid a", eggColor, "egg.");
      chuck();
    },
  };
})();

ChickenModule.layEgg();

///////////////

const myModule = (() => {
  const privateVar = "I am private";
  const privateMethod = () => "I am private method";

  return {
    publicVar: "I am public ",
    publicMethod: () => {
      console.log("i am calling private method", privateMethod());
    },
  };
})();

myModule.publicMethod(); // i am calling private method I am private m

// jquery uses the module pattern to protect internal details and reveal only the public api

const counter = (() => {
  let count = 1;
  return {
    getCount: () => count,
    increment: () => count++,
  };
})();

console.log("counter", counter);
console.log("counter", counter.getCount()); // 1
console.log("counter", counter.increment()); // 1
console.log("counter", counter.increment()); // 2
console.log("counter", counter.getCount()); // 3

# Singleton Pattern

The singleton assures only one instance of a class.

Real-world example

  • The redux store in react applications acts as a single, ensuring a singluar store instance. you can react more about in the docs.

  • Node.js module system caches modules, so repeated require calls for the same module don't recreate the module but return the cached version, acting like a singleton.

const ChickenForm = (() => {
  let instance;
  const createInstance = () => ({ totalChickens: 100 });
  return {
    getInstance: () => {
      if (!instance) {
        instance = createInstance();
      }
      return instance;
    },
  };
})();

console.log(ChickenForm);
console.log(ChickenForm.getInstance()); // { totalChickens: 100 }

class DatabaseConnection {
  constructor() {

# Obaserver pattern

The Observer pattern enables a subscription model where objects (observers) listen to events and get notified when events occur.

real-world-exmaples

javascript's event handling in browsers (DOM event listeners) in built upon the observer pattern.

Vuejs uses a reactive data model where component re-renders are triggered by data changes, which is an application of the observer pattern.

RxJs is library that provides utilities for working with observables, making extensive use of the observer pattern.

class Subject {
  constructor() {
    this.observers = [];
  }
  subscribe(fn) {
    this.observers.push(fn);
  }
  unsubscribe(fn) {
    this.observers = this.observers.filter((o) => o !== fn);
  }

  notify(data) {
    this.observers.forEach((fn) => fn(data));
  }
}

const observerfun1 = (data) => console.log("observer 1", data);
const observerfun2 = (data) => console.log("observer 2", data);
const observerfun3 = (data) => console.log("observer 3", data);

const subject = new Subject();
subject.subscribe(observerfun1);
subject.subscribe(observerfun2);
subject.subscribe(observerfun3);

console.log(subject.notify("click"));

# Registry Pattern

The registry pattern is a design pattern used to store and retrieve instances of objects.

It acts like a central place to manage the objects, and it is particularly useful when you want to access the instances from different parts of your application without needing to pass them around as parameters.
